# K-SafeGuard를 FR judge로 사용한 모델 평가

작성 2026-07-01 · 산출물 `results/model_metrics.csv` · `results/judge_agreement.csv` · 코드 [eval/run_eval.py](../../eval/run_eval.py) · [eval/eval_util.py](../../eval/eval_util.py)

## 1. 목적

새로 학습한 Moderator **K-SafeGuard**를 FR 평가의 **judge**로 도입하고, 기존 LLM judge(gpt-oss-safeguard-120b)와 비교한다. 통합 벤치(merged, 435)로 로컬 모델 10종의 응답을 재평가했다.

## 2. 방법

- **evaluator-type 분기 추가**: `eval/run_eval.py`에 `--evaluator_type {llm_judge, ksafeguard}`(auto 감지) 도입. `eval/eval_util.py`에 K-SafeGuard용 template·parser 추가.
  - K-SafeGuard: WildGuard 포맷으로 `(instruction, response)`를 입력받아 `Response refusal: yes/no`를 출력 → **refusal / compliance (2-way)**. gpt-oss는 `refusal / deflection / compliance (3-way)`.
- **출력 파일명에 evaluator 인코딩**: `results_<benchmark>_evaluator_<evaluator>.jsonl` / `metrics_..._<evaluator>.json` → 여러 evaluator 결과가 한 폴더에서 덮어쓰기 없이 공존.
- 지표: `behavior_acc`(정상 프롬프트 정상응답 + 유해 프롬프트 거부의 정확도), `safe_compliance`(정상 프롬프트 정상응답률 = 과잉거부의 역), `unsafe_refusal`(유해 프롬프트 거부율).

## 4. gpt-oss judge와의 비교

per-example **binary refusal 일치율 82.0%** (3565/4350). 혼동표(`results/judge_agreement.csv`):

| gpt-oss → K-SafeGuard | 수 |
|---|--:|
| refusal → refusal | 2139 |
| compliance → compliance | 1365 |
| compliance → refusal | 413 |
| deflection → refusal | 369 |
| deflection → compliance | 61 |
| refusal → compliance | 3 |

1. **refusal 판정은 거의 일치** — gpt-oss가 refusal로 본 2,142건 중 2,139건(99.9%)을 K-SafeGuard도 refusal로 판정. 반대 방향 오판(refusal→compliance)은 3건뿐.
2. **2-way이며 더 엄격한 성격** — gpt-oss가 compliance로 본 413건을 K-SafeGuard는 refusal로 판정. 경계성·hedged 응답을 refusal 쪽으로 더 엄격하게 본다.
3. **deflection을 대부분 refusal로 counting (의도된 설계)** — K-SafeGuard는 deflection이 없어, gpt-oss의 deflection 430건 중 369건(86%)을 refusal로 흡수한다. 회피·경고성 비응답을 과잉거부로 세는 것은 FR 취지에 부합하는 의도된 동작이다.
4. **모델별 랭킹은 대체로 일치** — 두 judge 모두 A.X·Midm·EXAONE-4.5를 상위, solar·HyperCLOVA를 하위로 매긴다.
   - 단, YanoljaNEXT-EEVE는 정상응답률(0.843)만 보면 과잉거부 최저 1위지만, **유해 프롬프트 거부율이 0.536으로 전 모델 중 최저**다(차하위 solar 0.776). 즉 이 모델은 유해 요청조차 잘 거부하지 못해 모든 요청에 응답하는 성향이라, 낮은 과잉거부가 장점이 아니므로 **논외**로 본다.

## 6. 한계

- K-SafeGuard는 2-way judge라 deflection을 별도 구분하지 않는다(설계상 refusal로 흡수). 3-way judge 대비 refusal(과잉거부)을 체계적으로 높게 보고하므로, **절대값 비교보다 동일 judge 내 상대 비교**에 적합하다.
- 세 벤치(merged·orbench·xstest)에서 경향이 일관되나, 모두 번역 기반 한국어 셋의 특정 분포다.
