# harmful contrast set 재구축 — 여러 harmful 데이터셋 + example 단위 유사도 매칭

작성 2026-07-07 · 출력 `dataset/final/merged_dataset_v2.jsonl` · **API 미사용(전부 오픈소스 로컬 모델)** · 번역 KJB 재사용 + instructTrans(vLLM) · 필터 gpt-oss-120b(vLLM) · 매칭 multilingual-e5-large(로컬)

## 1. 목적

기존 harmful(unsafe)은 **XSTest harmful을 그대로 번역**한 200개(+contrast_k_idioms 50)로, benign set과의
대응·대조가 약했다. 이를 **여러 harmful 데이터셋에서 가져와**, **benign 각 예시와 example 단위로 가장 유사한**
실제 유해 요청으로 교체한다. 핵심은 domain 균형이 아니라 **benign↔harmful 예시 간 유사도**다.

- **불가침**: `k_idioms`(safe 50)·`contrast_k_idioms`(unsafe 50) 그대로 유지. 일반 benign 135도 유지.
- **교체**: 일반 harmful contrast → 아래 파이프라인 산출 250개.
- **전 과정 오픈소스 로컬 모델만 사용(외부 API 금지).**

## 2. 파이프라인 (전부 로컬)

1. **오염 제거 소스 풀** `build_core_pool.py` — KJB 한국어(HarmBench/StrongREJECT/SALAD base) 재사용,
   AdvBench·JailbreakBench 제외(§3) + KS train 텍스트 오염 제거 + harm-domain 태깅 → core_pool 17,197.
2. **MaliciousInstruct(신규 소스) 로컬 재번역** — 원본 100개 EN을 **nayohan/llama3-instrucTrans-enko-8b(vLLM)**
   로 EN→KO(거부 0/100) → **gpt-oss-120b(vLLM) 필터** → 95개. (초기 gpt-oss 직접 번역은 97/100 거부하여 전용 번역기로 교체.)
3. **후보 선별 + 번역 품질 필터(gpt-oss-120b vLLM)** — batch1 260→260, batch2 356→322. 4개 소스 clean 풀 **578**.
4. **example 단위 유사도 매칭** `embed_match.py` — **intfloat/multilingual-e5-large(로컬, CPU)** 로 일반 benign
   132개(번역 아티팩트 3개는 앵커에서만 제외)와 clean 풀 578개를 임베딩, 각 benign의 최근접 harmful을
   라운드로빈(benign당 최대 3개)으로 250개 선별. 각 harmful은 **어떤 benign 예시의 의미적 최근접 이웃**.
5. **조립** `assemble_eval.py` — k_idioms 쌍·일반 benign 유지 + 매칭된 harmful 250 교체. 비파괴.

## 7. 한계 / 노트

- **번역기 혼재**: KJB(HarmBench/StrongREJECT/SALAD)는 KJB 파이프라인 번역 재사용, MaliciousInstruct는
