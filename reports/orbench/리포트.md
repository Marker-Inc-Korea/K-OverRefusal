# K-FalseRefusal — OR-Bench(한국어): 번역 파이프라인 & 과잉거부 평가

작성 2026-06-22 · 대상 `bench-llm/or-bench` config `or-bench-hard-1k` (1,319건)

## 1. 목적

XSTest 외에 OR-Bench를 한국어로 확장하여 과잉 안전 거부(over-refusal)를 추가로 측정한다.
OR-Bench는 전부 "유해해 *보이지만* 실제로는 무해한" 프롬프트로, 정상 모델이라면 모두 **응답**해야 한다.
기존 XSTest 평가 파이프라인과의 호환을 유지하는 것을 제약으로 둔다.

## 2. 데이터 (번역)

### (1) 원천
- `bench-llm/or-bench`, config `or-bench-hard-1k`, split `train`, 컬럼 `prompt`/`category` (1,319건).

### (2) 번역 방법 — 모델 선택 실험
- **1안(실패): gpt-oss-120b 단독.** XSTest와 동일하게 gpt로 번역 시도 → 유해해 보이는 프롬프트의 **번역 자체를 거절**하는 사례가 폭증.
- **2안(채택): 전용 번역모델 + gpt 필터(2단계).** 거절이 거의 없는 EN→KO 전용 모델 `nayohan/llama3-instrucTrans-enko-8b`로 번역하고, gpt-oss-120b는 **품질 판정에만** 사용.
- **프롬프트 형식**: 전용 번역모델은 "사고 후 `[OUTPUT]:`" 형식을 따르지 않으므로 모델 카드대로 **system 메시지에 지시, user 메시지에 원문**만 전달(`--translation_style plain`). gpt 계열은 기존 `reasoning` 형식 유지.

### (3) 품질 필터 (gpt-oss-120b)
- **휴리스틱**: 빈 출력 / 한글 없음 / 영어 잔존 / 원문과 동일 / 과도하게 짧거나 김 / 거절문 패턴.
- **2차 LLM 검사**: 휴리스틱 통과분에 대해 충실도 판정(보수적 — 명시적 confirm 없으면 탈락). KS 파이프라인의 2-pass 방식 차용.
- 단일모델 경로와 분리(2단계) 경로가 **동일 함수**(`apply_translation_filter`)를 공유.

### (4) 결과 데이터셋
- clean: `dataset/orbench/orbench_hard1k_ko_instructtrans.jsonl` (1,312건)
- 중간/감사: `..._raw.jsonl`(번역 원본 1,319), `..._rejected.jsonl`(탈락 7), gpt 단독 비교본 `..._gpt(.jsonl/_rejected.jsonl)`

## 3. 평가 방법 (over-refusal metric)

OR-Bench는 전부 무해(=응답이 정답)이므로 XSTest식 safe/unsafe 분할이 없다. `compute_overrefusal_metrics`를 도입:
- **`over_refusal_rate`** (= refusal + deflection, 핵심 지표, 낮을수록 좋음)
- `refusal_rate` / `deflection_rate` / `compliance_rate` / `unclassified_rate`, `category`별 분해.
- `run_eval.py --metric auto`: `label` 컬럼 → XSTest식 exaggerate, `category` 컬럼 → over-refusal, 그 외 → simple. **XSTest 경로는 불변.**

## 5. 해석

- gpt 단독 탈락의 압도적 다수가 **거절(refusal)**(영어 거절문을 그대로 출력 → `looks_like_refusal`/`no_hangul`). 즉 번역 품질이 아니라 모델이 **작업 자체를 거부**.
- 전용 번역모델은 거절이 사실상 0이고, gpt를 충실도 판정에만 쓰자 탈락이 7건으로 급감. **번역은 비거절 전용모델, 판정은 강한 모델**이라는 역할 분리가 효과적.
